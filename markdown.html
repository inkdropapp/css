Migration Plan: Webpack + Grunt → electron-vite

Goal

Replace the dual Webpack (bundling) + Grunt (orchestration) build system with a single modern toolchain built on Vite. The user's instinct ("Vite is right, but what replaces Grunt? Maybe Vite+?") is correct — and the precise answer is electron-vite, which subsumes both Webpack and Grunt: it bundles the main + renderer processes with Vite/Rollup and orchestrates the dev/build lifecycle (watch, hot-reload, asset copy) that Grunt currently hand-rolls.

electron-builder is kept as-is — it already consumes build/ and is well past Grunt's responsibilities. This migration only replaces the pre-packaging pipeline.

The one constraint that shapes everything: nodeIntegration

This is not a vanilla web renderer. Verified facts from the codebase:

  • Windows run with contextIsolation: false, nodeIntegration: true, nodeIntegrationInWorker: true (src/main/utils/window-manager.ts:213).
  • The renderer directly imports Node/native/Electron modules at runtime:
    • electron (ipcRenderer / webFrame / shell) — 22 files
    • @inkdropapp/pouchdb-store — 9 files (PouchDB runs in the renderer)
    • @inkdropapp/fts-store — 6 files (full-text search in the renderer)
  • fs, path, os, module, node:fs, node:stream, node:querystring
  • All package.json dependencies are externalized in both Webpack configs and resolved at runtime via Node require() from node_modules (symlinked into build/ in dev, packed into the asar in prod). devDependencies are the ones actually bundled. This inverted convention is deliberate.
  • The renderer is emitted as a CommonJS module (library.type: 'commonjs2') and injected by a hand-written bootstrap in src/index.html (<script src="browser-main.js"> + main.css), loaded over file:// with query-string params (windowId, entry, dark-ui, runCommands…).

⚠️ electron-vite officially does not support nodeIntegration — its happy path is contextIsolation: true + preload + an ESM/browser renderer. Because Inkdrop runs PouchDB + FTS + fs directly in the renderer, removing nodeIntegration is a data-layer rewrite touching 400+ files and is explicitly OUT OF SCOPE. The plan therefore keeps nodeIntegration and configures the renderer to externalize Node built-ins + electron + runtime dependencies, emitting CommonJS — a well-trodden (if off-the-happy-path) electron-vite setup using vite-plugin-commonjs-externals.

This risk is real but bounded, which is why Phase 0 is a throwaway spike to prove the nodeIntegration renderer boots under electron-vite before any real migration work.

Why electron-vite over the alternatives:

  • electron-vite (chosen): Purpose-built three-target (main/preload/renderer) orchestrator. Its externalizeDepsPlugin externalizes dependencies and bundles the rest by default — a 1:1 match for the existing convention. Single electron.vite.config.ts. Gives main hot-reload + renderer HMR for free. Wires ELECTRON_RENDERER_URL (dev server) vs loadFile (prod) automatically.
  • Alternative — plain Vite + vite-plugin-electron + npm-run-all: More manual; you reimplement the orchestration electron-vite gives for free. Keep as fallback if electron-vite's opinions fight the nodeIntegration setup harder than the Phase 0 spike predicts.
  • Rejected — Electron Forge + @electron-forge/plugin-vite: Forge wants to own packaging too, displacing the heavy, working electron-builder investment (signing, notarization, S3 publish, flatpak, NSIS). Too much blast radius.

Motivation / payoff — what electron-vite optimizes here

Biggest win: a real dev loop

Today npm run dev (grunt) symlinks node_modules/packages into build/, copies assets, and runs webpack --watch for main + renderer. The renderer loads from file://…/index.html with no dev server and no HMR — so editing any of the 402 renderer files means webpack rebuild → manual window reload → lost app state (re-navigate to the note, re-trigger the test).

electron-vite replaces that with:

  • Renderer HMR — edit a component, update without a full reload or losing state. The single biggest QoL change.
  • Main-process auto-restart — replaces the manual Electron restart after shell:watch-main.
  • Rust-fast rebuilds — ts-loader + Webpack → esbuild/SWC transpile + Rolldown (Vite 8).

⚠️ Honesty caveat: the nodeIntegration + CJS-externals renderer is the one case where Vite HMR is least guaranteed. Worst case = fast rebuild + auto-reload (still better than today); best case = full HMR. Phase 0 confirms which.

Representative config (target shape, not final)

electron-vite config (sketch)
electron.vite.config.ts 
import { resolve } from 'path'
import { defineConfig, externalizeDepsPlugin } from 'electron-vite'
import react from '@vitejs/plugin-react-swc' // Rust/SWC, Vite 8-ready
import commonjsExternals from 'vite-plugin-commonjs-externals'
import { viteStaticCopy } from 'vite-plugin-static-copy'

const settings = require('./settings.js')[process.env.TARGET || 'development']
const define = {
  _settings: JSON.stringify(settings),
  _DEMO_MAX_NOTE_COUNT: JSON.stringify(5)
}
const alias = { '@': resolve('src') }

export default defineConfig({
  main: {
    plugins: [externalizeDepsPlugin()],
    define,
    resolve: { alias },
    build: {
      lib: { entry: 'src/main.js' },
      rollupOptions: { output: { banner: 'global.require = require;' } },
      sourcemap: process.env.TARGET?.startsWith('production') ? false : true
    }
  },
  renderer: {
    root: 'src',
    define,
    resolve: {
      alias,
      mainFields: ['browser', 'module', 'jsnext:main', 'main']
    },
    plugins: [
      react(),
      externalizeDepsPlugin(),                 // externalize runtime `dependencies`
      commonjsExternals({                       // keep nodeIntegration `require()` working
        externals: ['electron', /^node:/, 'fs', 'path', 'os', 'module']
      }),
      viteStaticCopy({ targets: [
        { src: '../packages', dest: '.' },
        { src: '../exports', dest: '.' },
        { src: '../images/**/*', dest: '.' },
        { src: 'browser/components/fonts/*.woff', dest: 'fonts' }
      ]})
    ],
    build: {
      rollupOptions: {
        input: { index: resolve('src/index.html') },
        output: { format: 'cjs' }
      },
      sourcemap: process.env.TARGET?.startsWith('production') ? false : true
    }
  }
})

Two problems found & solved (this is why Phase 0 exists)

  1. vite-plugin-commonjs-externals (the sketch's choice) is incompatible with Vite 8 / Rolldown. It did not intercept node builtins — Vite browser-externalized fs/path/electron into empty stubs → path.join is not a function at runtime.
    • Fix (plan correction): use vite-plugin-electron-renderer instead. It auto-handles electron + node builtins and loads node-only/native deps via require() (declare them { type: 'cjs' }). This supersedes every vite-plugin-commonjs-externals reference in the sketch/mapping above. Electron 41 ≥ 35, so runtime require() is supported, and no forced output.format: 'cjs' was needed — the default HTML/ESM renderer build worked.
  2. import.meta.dirname comes through undefined when electron-vite bundles a .mts config → relative paths broke. Fix: anchor paths to process.cwd().

Raw-TS @inkdropapp packages — BUNDLE them (corrected, verified)

The first spike attempt externalized @inkdropapp/pouchdb-store / fts-store and hit Node's Stripping types is currently unsupported for files under node_modules (they ship "main": "src/index.ts"). That approach was wrong — corrected below.

Correct model (matches webpack): these two packages are in the app's devDependencies, and webpack's externals only covers Object.keys(dependencies) — so webpack bundles them and ts-loader transpiles their .ts. They drag in a subtree of other raw-TS @inkdropapp/* packages (model-private, search-utils, protocol-utils, logger, …).

Verified: importing them normally (NOT externalized, NOT declared in the electron-renderer plugin) makes Rolldown bundle + transpile the whole raw-TS subtree — build transformed 94 modules → 821 KB, no Stripping types error, and at runtime createInkdropDB, InkdropDatabase, and InkdropFTSStore all resolve as functions. So Rolldown/oxc handles raw-TS node_modules packages just like ts-loader. ✅

compile-cache is for PLUGINS, not these packages. @/shared/modules/compile-cache (CompileCache.install(...) in environment.tsx) is the runtime TS/JSX transpile hook for installed plugins (packages/ + user plugins) — unrelated to the @inkdropapp store packages, which are build-time bundled. No change needed there for these deps.

Dependency-split rule to carry into Phase 1: dependencies → externalized / runtime require (react, better-sqlite3, ajv, event-kit, inkdrop-model, @napi-rs/keyring, …); everything else imported, including the raw-TS @inkdropapp devDeps → bundled + transpiled by Rolldown. (better-sqlite3 has a normal JS entry and is the native-dep proxy here.)

✅ Phase 4 — Results (2026-06-07) — plugin loading works under electron-vite dev; out/ app root assembled

The core question answered: CJS plugins keep working, require stays alive in the renderer

Plugins are not part of the Vite graph — neither Webpack's before nor Vite's now. require exists in the renderer because of Electron (nodeIntegration: true / contextIsolation: false), independent of the bundler. Plugins are loaded at runtime by Node's CJS loader via the custom .js/.jsx/.es6 handlers compile-cache.ts installs on require.extensions (transpile → module._compile). Vite only bundles src/. The ESM app + CJS plugins meet at one CJS bridge — exports/inkdrop.js = module.exports = window.inkdrop.getExports(). No plugin conversion needed.

🐛 Post---dir correction — the production renderer was actually broken (bare external imports)

The --dir check above exercised the main process + data layer (which load fine — app src/ chunks DO get the require rewrite) but never rendered the UI, so it didn't load the lazy node_modules chunks where the bug lived. User hit it on a real pn build:mac launch: Uncaught TypeError: Failed to resolve module specifier "react".

Root cause: under Rolldown, bundled node_modules deps (lucide, glob, framer-motion, @lezer/common, …) are resolved/bundled by Rolldown's native path, which bypasses JS plugin resolveId/transform hooks — so vite-plugin-electron-renderer's import→__require rewrite only reaches app src/. Those dep chunks keep bare import … from "react" / "node:fs" / "inkdrop-model", and the renderer's browser ESM loader (file://, type="module") can't resolve bare specifiers → throws at load. (Dev hides it: Vite's dev server resolves bare specifiers; and src/'s react still routes through require so the singleton/plugin-sharing premise is intact.)

Fix: added a renderChunk post-pass requireBareImports() to the renderer config (enforce: 'post', apply: 'build') that parses each output chunk and rewrites every remaining bare-specifier static import to require(...) with the plugin's exact interop (default → ?.default ?? mod, namespace → mod, named → mod[key]; node:* kept as requires, URL schemes skipped). Verified via electron-vite preview: no module-specifier error, renderer loads, and FTS runs live SELECT … FROM notes_fts MATCH … queries. (Need to re-run pn build:mac to bake the fix into the package.)
